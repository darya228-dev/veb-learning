# Сценарій A — SQL Injection

## Було

У початковій версії застосунку в частині роботи з користувачами SQL-запити могли формуватися через конкатенацію рядків. Наприклад, значення `id`, отримане від користувача, могло напряму вставлятися у SQL-запит.

Небезпечний підхід виглядає так:

```ts
db.get(`SELECT * FROM users WHERE id='${id}'`, ...)
```

У такому випадку введення користувача стає частиною SQL-команди. Це створює ризик SQL Injection, оскільки зловмисник може передати спеціальний рядок, який змінить логіку запиту.

## Відтворення проблеми

Для перевірки було використано запит:

```powershell
curl.exe "http://localhost:3000/api/v1/users/1'%20OR%20'1'%3D'1"
```

Такий ввід імітує спробу SQL Injection через умову:

```sql
' OR '1'='1
```

Якщо запит формується небезпечно, така умова може призвести до повернення зайвих даних.

## Виправлення

Для виправлення SQL Injection було використано параметризовані запити. Тепер користувацьке значення не підставляється напряму в SQL-рядок, а передається окремим параметром.

Безпечний варіант:

```ts
db.get("SELECT * FROM users WHERE id = ?", [id], ...)
```

Також параметризовано створення та видалення користувачів:

```ts
db.run("INSERT INTO users (id, name) VALUES (?, ?)", [id, name], ...)
db.run("DELETE FROM users WHERE id = ?", [id], ...)
```

## Як працює захист

Параметризований запит відокремлює SQL-команду від даних користувача. Це означає, що навіть якщо користувач введе рядок типу:

```sql
' OR '1'='1
```

SQLite сприйме його не як частину SQL-команди, а як звичайне текстове значення параметра `id`.

## Перевірка після виправлення

Після виправлення команда:

```powershell
curl.exe "http://localhost:3000/api/v1/users/1'%20OR%20'1'%3D'1"
```

повернула:

```json
{"status":404,"code":"NOT_FOUND","message":"User not found","details":[]}
```

Отже, SQL Injection не спрацював, а небезпечний ввід був оброблений як звичайний параметр.

# Сценарій Б — XSS

## Було

У frontend-частині застосунку дані користувача могли відображатися в таблиці через `innerHTML`. Це небезпечно, тому що браузер може сприйняти введений текст як HTML або JavaScript-код.

Небезпечний підхід:

```js
tableBody.innerHTML = state.items.map(item => `
  <tr>
    <td>${item.subject}</td>
  </tr>
`).join("");
```

Якщо користувач введе HTML/JS-код, він може бути виконаний у браузері.

## Відтворення проблеми

Для перевірки можна було створити заявку з таким текстом:

```html
<img src=x onerror=alert('XSS')>
```

Якщо дані вставляються через `innerHTML`, браузер може інтерпретувати цей рядок як HTML і виконати JavaScript у події `onerror`.

## Виправлення

Для виправлення XSS рендеринг таблиці було змінено. Замість вставлення HTML-рядків використано створення DOM-елементів через:

```js
document.createElement()
```

А користувацькі значення вставляються через:

```js
textContent
```

Приклад безпечного відображення:

```js
const td = document.createElement("td");
td.textContent = item.subject || "";
tr.appendChild(td);
```

Також для перегляду опису заявки було додано окреме модальне вікно. Опис не вставляється в таблицю, тому довгий текст не ламає структуру інтерфейсу. Дані в модальному вікні також вставляються через `textContent`.

## Як працює захист

`textContent` не інтерпретує текст як HTML. Якщо користувач введе:

```html
<img src=x onerror=alert('XSS')>
```

браузер покаже цей рядок як звичайний текст, а не виконає його як код.

## Перевірка після виправлення

Після виправлення HTML/JS-ввід відображається у таблиці або в модальному вікні як звичайний текст. Вікно `alert` не відкривається, JavaScript не виконується.

# Сценарій В — Broken Access Control / IDOR

## Було

У застосунку кожна заявка має поле `userId`, тобто належить певному користувачу. Але якщо доступ до заявки перевіряється тільки за її `id`, виникає ризик IDOR.

IDOR означає, що користувач може підставити чужий ідентифікатор ресурсу в URL і спробувати отримати доступ до чужих даних.

Наприклад:

```txt
/api/v1/tasks/1
```

Якщо заявка `1` належить іншому користувачу, звичайний користувач не повинен мати до неї доступ.

## Відтворення проблеми

Було виконано запит від імені користувача `2` до заявки `1`:

```powershell
curl.exe -H "X-Demo-UserId: 2" http://localhost:3000/api/v1/tasks/1
```

До виправлення така ситуація могла призвести до отримання чужої заявки.

## Виправлення

Було додано middleware `currentUser`, яке визначає поточного користувача через заголовок:

```txt
X-Demo-UserId: <id>
```

Middleware перевіряє, чи існує такий користувач у таблиці `users`. Якщо заголовка немає або користувач не існує, сервер повертає `401 UNAUTHORIZED`.

Також було додано роль користувача:

* `admin`;
* `user`.

У сервісі заявок реалізовано перевірку доступу:

```ts
function checkOwnerOrAdmin(task, currentUserId, currentUserRole) {
  if (currentUserRole === "admin") return;

  if (task.userId !== currentUserId) {
    throw new ApiError(403, "FORBIDDEN", "You do not have access to this task");
  }
}
```

Ця перевірка застосовується до чутливих операцій:

* читання заявки;
* редагування заявки;
* видалення заявки.

## Як працює захист

Користувач не може самостійно вказати в тілі запиту, ким він є. Поточний користувач визначається на backend через заголовок `X-Demo-UserId`.

Після цього backend перевіряє:

1. Чи існує користувач.
2. Яка в нього роль.
3. Чи належить заявка цьому користувачу.
4. Якщо користувач має роль `admin`, він може бачити всі заявки.
5. Якщо користувач звичайний, він може бачити тільки власні заявки.

Таким чином, frontend не “ховає” чужі заявки самостійно. Основна перевірка виконується саме на backend.

## Перевірка після виправлення

Команда:

```powershell
curl.exe -H "X-Demo-UserId: 2" http://localhost:3000/api/v1/tasks/1
```

повернула:

```json
{"status":403,"code":"FORBIDDEN","message":"You do not have access to this task","details":[]}
```

Це означає, що звичайний користувач не може отримати доступ до чужої заявки.

Перевірка адміністратора:

```powershell
curl.exe -H "X-Demo-UserId: 1" http://localhost:3000/api/v1/tasks
```

Адміністратор отримує всі заявки, оскільки має роль `admin`.

# Сценарій Г — Security Misconfiguration

## Було

У початковій конфігурації застосунку не було достатнього набору базових security headers. Також було важливо зробити єдиний формат помилок і обмежити CORS.

Без таких налаштувань застосунок може мати слабший рівень захисту, а клієнт може отримувати зайві технічні деталі.

## Виправлення

До `app.ts` було додано базові security headers:

```txt
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: no-referrer
Content-Security-Policy: default-src 'self'
```

Також налаштовано CORS тільки для локального frontend:

```txt
http://localhost:5173
http://127.0.0.1:5173
```

Додатково дозволено заголовок:

```txt
X-Demo-UserId
```

Помилки повертаються в єдиному форматі:

```json
{
  "status": 403,
  "code": "FORBIDDEN",
  "message": "You do not have access to this task",
  "details": []
}
```

Для внутрішніх помилок сервер не повертає stack trace користувачу, а показує загальне повідомлення:

```json
{
  "status": 500,
  "code": "INTERNAL_ERROR",
  "message": "Internal server error",
  "details": []
}
```

## Як працює захист

Security headers додають базовий рівень захисту браузера:

* `X-Content-Type-Options: nosniff` забороняє браузеру самостійно вгадувати тип контенту;
* `X-Frame-Options: DENY` забороняє відкривати сторінку у frame/iframe, що зменшує ризик clickjacking;
* `Referrer-Policy: no-referrer` не передає адресу попередньої сторінки;
* `Content-Security-Policy: default-src 'self'` обмежує джерела завантаження ресурсів.

CORS обмежує, з яких адрес frontend може звертатися до API. Замість дозволу для всіх адрес використано тільки локальні адреси навчального frontend.

## Перевірка після виправлення

Було виконано команду:

```powershell
curl.exe -i -H "X-Demo-UserId: 1" http://localhost:3000/api/v1/tasks/stats
```

У відповіді сервера присутні такі заголовки:

```txt
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: no-referrer
Content-Security-Policy: default-src 'self'
```

Отже, мінімальний hardening застосунку виконано.

# Додатково реалізовано

У межах лабораторної роботи додатково було реалізовано такі покращення:

1. Вибір поточного демо-користувача на frontend.
2. Рольова модель: `admin` та `user`.
3. Адміністратор бачить усі заявки.
4. Звичайний користувач бачить тільки власні заявки.
5. Агрегація заявок підлаштовується під поточного користувача.
6. Статистика користувачів показує, скільки заявок створив кожен користувач.
7. Додано сортування таблиць за кліком на назву стовпця.
8. Додано окремий перегляд опису заявки через кнопку «Опис».
9. Опис заявки відображається в модальному вікні й не ламає структуру таблиці.
10. Для відображення користувацьких даних використовується `textContent`, тому XSS-захист зберігається.
